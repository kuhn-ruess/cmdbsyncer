# Version 3.12


## Unreleased
- FEAT: Admin list/detail tables — the table header stays pinned at the top of the viewport (just below the fixed navbar) when the list is scrolled, so column labels remain visible no matter how far down you scroll a long host / rule / account table. `position: sticky; top: 60px;` on `.table thead th`, with the `.table-responsive` wrapper switched from `overflow: hidden` to `overflow: visible` — sticky requires every ancestor up to the scroll container to keep vertical overflow `visible`. Modal and `.table-responsive` tables pin to their local top (0) instead of the page navbar
- FEAT: Host list — full-text search across hostname AND any label value. The top-right search box now runs a single Mongo `$or` with `$regex` on hostname and `$expr` / `$objectToArray` / `$regexMatch` over every `labels` value, so a query like `nginx` or `FRA-DC1` hits hosts that carry that string anywhere without opening the multi-step filter dialog
- FEAT: Host list — new bulk actions "Bulk Edit Labels" (add / overwrite / remove / rename a label across all selected hosts in one modal) and "Copy as new" (clone hostname-suffix-copy with all labels, cmdb_fields and cmdb_templates; timestamps reset). Both open as modals on top of the list, the bulk-label modal has one form with a radio switch for the action. Row action: new copy-icon + new history-icon per host row. All three new actions ("Bulk Edit Labels", "Copy as new", "Set Template") are only surfaced when `CMDB_MODE=True` so import-only installations keep a clean dropdown
- FEAT: Host detail view — labels and inventory now render in the same compact grid style as the edit form. Each label row carries a BSON-type badge (so a string `"True"` stays visually distinct from bool `True`) plus a source badge (either `manual` or the assigned template name). Inventory uses the same row style but single-column so long values (UUIDs, firmware strings) don't get truncated. Log entries render as a compact scrollable list with timestamps split out from the message. The separate "CMDB" block is gone — every template-derived label already carries its template-name badge inline with the manual labels
- FEAT: Per-host label history / timeline — a new `host_label_change` collection records every `add` / `update` / `remove` on a host's labels (with old/new value, source `manual` / `import` / `template`, acting user email, timestamp). A row action on every host opens `/admin/host/timeline?obj_id=…` with a grouped-by-day view and colour-coded change badges. The Enterprise Audit Log also receives a single `host.label.changed` event per save (changes field carries the diff), so compliance installs see label mutations without enabling the per-save signal on Host itself (which would be millions of writes per sync)
- FEAT: Debug page overhaul — `/admin/host/debug` surfaces what actually matters first: **Outcomes** (what will fire) and **Filtered Labels** (the attribute set after filter + rewrite rules) appear side-by-side at the top, always expanded. Custom Attributes get their own block only when present. The raw attribute list is collapsed into a `<details>` at the bottom. The Rules accordion grew hit-count badges (`3/14`) per rule group, treffer rows are green-tinted with check / times icons, and every rule name links directly to the admin edit page for that rule
- FEAT: Debug page supports five modes — Checkmk Host, Netbox Device, Ansible Host, i-doit Host, VMware Host. Checkmk now additionally debugs the "Setup Rules" engine (CheckmkRuleMngmt); Netbox additionally debugs VM Attributes and Cluster Attributes rules. Rule-name deep-links work for every mode (mapped to the Flask-Admin default endpoint `model.__name__.lower()`)
- FIX: Debug page rule-name links used to always render `href="#"` except for CustomAttributes — now every known rule group carries a real deep link to the admin edit page, and rows with no URL render as plain text instead of a dead hash link
- FIX: Host debug icon changed from 🚀 `fa-rocket` to 🐛 `fa-bug` — a bug is the standard debug affordance and makes it instantly clear what the icon does
- FEAT: CMDB host edit form split into clearly-named sections — **Object** (hostname + available), **Manual Labels (editable)** with a help text explaining that this list is the single source of truth (labels seeded from imports land here too), and **Template Labels (read-only)** showing the assigned templates and their merged labels separately. No more mixing the two kinds of labels in the same list
- FEAT: CMDB `cmdb_fields` editor is now a compact two-column grid on wide screens (single column under 992 px) — the per-row card is flattened, the redundant "Manual Labels #N" caption hidden, the value input grows to fill the row width, and entries are auto-sorted alphabetically on both render and save, so there is no drag-to-reorder UI to worry about
- FEAT: In the host create/edit form, `Object Type` is no longer selectable — every host created or edited through this view is pinned to `host` (the field is hidden on the form and re-asserted in `on_model_change`)
- FIX: Unauthenticated top navbar no longer shows a "Request new Password" link
- FEAT: Enterprise 4-Eyes Approval Workflow (license feature `approval_workflow`) — configurable per-resource-type gate that queues mutations as `PendingChange` rows until a second admin approves. Available for `Account`, `User`, `CronGroup`, `Rule`, `CustomAttributeRule`, `SecretStore`, `AccountSecretBinding`, `WebhookPolicy`, `NotificationChannel`, `NotificationRule`, `BackupConfig`, `AuditSink`. Requester-cannot-approve-own is hard-enforced (same-user submissions are silently skipped in the Approve/Reject bulk actions). Admin UI under "Approvals" with Approval Policies (pick the types to gate) + Pending Changes (read-only queue with side-by-side diff of only the fields that changed, secrets redacted in display). Apply on approval uses raw pymongo `replace_one`/`insert_one`/`delete_one` so nested EmbeddedDocuments/ReferenceFields/DictFields round-trip byte-for-byte. Every submit/approve/reject is recorded in the audit log (`approval.submitted`, `approval.approved`, `approval.rejected`, `approval.apply_failed`) and routed into the notification dispatcher (when both features are co-licensed) so #approvers gets review requests and requesters get decision outcomes. Sensitive fields (`password`, `password_crypted`, `webhook_token`, `master_password_env`, `vault_token_env`, `tfa_secret`, `signing_secret`) are masked in the diff view but the real value stays in the payload so password rotations actually apply on approval. Short-TTL policy cache (5 s) means enabling/disabling a policy takes effect cluster-wide without a restart
- FEAT: Enterprise Scheduled Backups (license feature `scheduled_backup`) — encrypted, rotated `mongodump` backups streamed to any S3-compatible object storage (AWS S3, GCS, Azure Blob, MinIO, Backblaze B2, Wasabi, DigitalOcean Spaces) via a single boto3 codepath. Pipeline: `mongodump --archive → gzip → [gpg --encrypt] → S3 PutObject`, streaming end-to-end so a 10 GB dump never touches disk. Per-config retention automatically deletes backups older than `retention_days` after each upload. New `enterprise-backup_run` cronjob task lets operators schedule backups through the existing CronGroup mechanism. Admin UI under "Backups": editable Configs + read-only History (size, duration, S3 key, error). Restore via CLI `cmdbsyncer enterprise backup restore <config> <key>`. Every run emits `backup.succeeded` / `backup.failed` audit + notification events
- FEAT: Enterprise Prometheus Metrics (license feature `prometheus_metrics`) — `/metrics` endpoint in Prometheus text format for scraping by Prometheus / VictoriaMetrics / Grafana / Datadog / New Relic. Metrics generated on-scrape from live Mongo state (no in-process counters to persist, HA-safe): `cmdbsyncer_info` with license metadata, per-CronGroup `enabled` / `running` / `failure` / `last_start_timestamp_seconds` / `last_success_timestamp_seconds` / `last_end_timestamp_seconds` / `last_duration_seconds` / `next_run_timestamp_seconds` gauges, `cmdbsyncer_hosts_total` + `cmdbsyncer_hosts_stale_24h_total`. Optional Bearer-token auth via `PROMETHEUS_METRICS_TOKEN_ENV` env-var name (fail-closed: env var set but empty returns 503). Zero runtime dependency (text format written by hand, no `prometheus_client` pull-in). CSRF-exempted blueprint
- FEAT: Enterprise Webhook Signatures (license feature `webhook_signatures`) — hardens the `/api/v1/syncer/cron/trigger/<group>` endpoint with cryptographic request signing (HMAC-SHA256 over `<timestamp>.<body>`, Stripe/GitHub-style), replay-protection via configurable timestamp window (default 300s), and per-group IP allowlists (IPv4/IPv6 CIDRs). Enforced only on CronGroups that have an enabled `WebhookPolicy` attached, so the feature can roll out group-by-group without breaking existing token-auth integrations. Admin UI under "Cronjobs → Webhook Policies" with auto-generated signing secret and a "Regenerate signing secret" bulk action for rotation. Validation outcomes route into the audit log (`webhook.triggered`/`webhook.rejected` with exact reason) and the notifications pipeline. IP allowlist is checked before signature so attackers cannot probe source-IP restrictions by flipping payloads
- FEAT: Enterprise Notification Routing (license feature `notifications`) — active alert pipeline that dispatches internal events to Slack, MS Teams, email, and generic HTTPS webhooks with rule-based routing, Jinja-templated payloads, per-dedup-key cooldowns and per-rule hourly rate limits. Wired sources: cron group failures/recoveries, webhook rejections, login failures/successes, CRUD on audited models (when `audit_log` is co-licensed). Admin UI under "Notifications" with Channels (plus a "Send test notification" bulk action) and Rules (regex matchers on event/source/target, severity gate). Slack: Block Kit with severity-coloured headers; MS Teams: Adaptive Cards; email: reuses the OSS SMTP config; webhook: HMAC-SHA256-signed JSON POST with env-var secret. Dispatch runs on a background thread with a bounded queue so a slow Slack endpoint never back-pressures login or cron ticks
- FEAT: Enterprise Audit → SIEM Streaming (license feature `audit_siem_streaming`, requires `audit_log`) — fans every persisted `AuditEntry` out in real time to Splunk HEC, syslog over TCP/TLS (RFC5424 framing), or a generic HTTPS webhook so compliance teams get a tamper-evident, write-only copy outside the syncer database. Admin UI under "Audit Log → SIEM Sinks"; delivery is asynchronous via a background worker so SIEM outages never back-pressure audit writes. All tokens (Splunk HEC token, Bearer token) come from env vars — never persisted in the DB
- FEAT: Enterprise Native OIDC Login (license feature `oidc_login`) — logs users in directly against Azure AD / Entra ID, Okta, Keycloak, Google Workspace, Auth0 or any OIDC-compliant IdP without needing a reverse-proxy with `mod_auth_openidc` in front. Discovery-URL based config (`OIDC_ISSUER`), client secret via env var (`OIDC_CLIENT_SECRET_ENV`), group claim → role mapping with the same union-of-groups semantics as LDAP, optional required-group gate, auto-create of local users on first login. Routes `/oidc/login` and `/oidc/callback` are added by a blueprint only when the license is active; every login attempt (success / token-exchange-failure / required-group-missing / no-email / auto-create-off) is recorded in the audit log when `audit_log` is co-licensed
- FEAT: Enterprise Audit Log (license feature `audit_log`) — append-only, WORM-style compliance trail under a new "Audit Log" admin category. Automatically records every login (success / failure with reason / logout), webhook trigger (accepted / rejected with source IP), and CRUD on Accounts, Users, Cron Groups, Rules, Custom Attribute Rules, Config, Secret Stores and Account Bindings — with field-level before/after diffs. Sensitive values (`password`, `password_crypted`, `webhook_token`, `master_password_env`, `vault_token_env`, `tfa_secret`) are stored as `***REDACTED***` in the diff; high-churn status fields (`last_start`, `last_ended`, `last_success_at`, `next_run`, `is_running`, `pid`, `last_message`, `all_messages`, `last_login`, `run_once_next`) are excluded so the signal-to-noise ratio stays high. View is read-only (no create/edit/delete), exports as CSV and JSON for evidence packs. Retention is a dedicated bulk action "Prune older than 365 days" which itself records an audit entry. MongoEngine-signal wiring means plugins get auto-audited without per-plugin code; custom events can still be emitted via `application.helpers.audit.audit(event_type, **context)`. Trace IDs are pulled from the same headers used by `json_logging` so audit and log streams correlate. Community installs never see the menu or schema because the module registers itself only when the license activates the feature
- FEAT: Enterprise JSON / ECS Logging (license feature `json_logging`) — replaces the plain-text log stream with one-line Elastic Common Schema (ECS 8.11) JSON on stdout, ready for Loki / OpenSearch / Elastic / CloudWatch / Datadog / Splunk / Fluent Bit / Vector without any parser mapping; emits `service.*`, `host.*`, `process.*`, `event.*` (source, outcome, details), Flask request context (`url.path`, `http.request.method`, `client.ip`), trace IDs auto-extracted from common headers (`X-Request-ID`, `X-Cloud-Trace-Context`, `X-Amzn-Trace-Id`, `traceparent`), exception info via `error.type`/`error.stack_trace`; service identity picked up from `OTEL_SERVICE_NAME`, `DEPLOYMENT_ENVIRONMENT`, `CMDBSYNCER_VERSION`, `AWS_REGION`; community installs never see the feature because the configurator is only registered when the license activates it
- FEAT: Enterprise Secrets Manager (license feature `secrets_manager`) — account passwords can be bound to an external vault (KeePass `.kdbx` file in v1; more providers planned) instead of being stored in the syncer DB; binding is configured in a new "Secrets Manager" admin category with Secret Stores + Account Bindings; resolution happens transparently in `get_account_by_name` so all existing plugins benefit with no plugin changes; master passwords are pulled from named env vars, never persisted; community installs never see the menu or schema because the module registers itself only when the license activates the feature
- FEAT: Cron groups can be triggered via webhook — enabling `webhook_enabled` on a group auto-generates a per-group `webhook_token`, and external systems (GitHub, Jenkins, Netbox, …) can then `POST /api/v1/syncer/cron/trigger/<group>` with an `X-Webhook-Token: <token>` header to fire an out-of-schedule run on the next cron pass. Normal user auth still works on the same endpoint, and a "Regenerate Webhook Token" bulk action rotates the secret if it leaks. Token comparison uses `hmac.compare_digest` to avoid timing-based discovery
- FEAT: Cron groups survive crashes and surface monitorable state — a new per-group `continue_on_error` flag lets the remaining tasks of a group run after a task fails (default off keeps the original "stop on first error" behaviour); stale `is_running` locks left behind by killed processes (OOM, container restart, `kill -9`) are released automatically on the next pass when the recorded PID no longer exists; `CronStats` records `last_success_at` and `pid` so external monitoring can alert on "last successful run older than N minutes"; the lock release moved into `try/finally` so a `KeyboardInterrupt` can no longer leave a group marked running
- FEAT: Admin UI refresh — modernized login, start page, top menu bar and list/detail tables; shared card/hero look across logout, 2FA, license, password forms, bulk-action confirmations and the changelog archive
- FIX: Host list "Label Key:Value" and "Inventory Key:Value" filters now match boolean and numeric labels (`input_monitoring:True`, `port:443`) — BSON booleans/ints are not string-matchable with `$regex`, so the filter additionally does an equality match against the typed value. The filter now also accepts real regex in the value part (the filter operation is advertised as "regex search"), so `:.*` matches everything and `:web-\d+` works as expected
- FIX: Host list label preview no longer stretches the row when labels have very long values — each badge is capped with ellipsis truncation (full value on hover) and the overall cell has a width bound, so a single 500-char label value can't push the table off-screen
- FIX: Host list "CMDB Attributes" and "CMDB" (template) columns apply the same width/ellipsis cap to their value badges, so long CMDB field values no longer push the row off-screen
- FEAT: Host detail view shows a Type column next to Labels, Inventory and CMDB values, so admins can tell a string `"True"` apart from a BSON bool `True` (they match different filters) without having to inspect the raw document
- FIX: MSSQL/ODBC importer with `allow_ddl` no longer aborts with `pypyodbc.Error HY010 Function sequence error` — the connection is now committed after the SELECT has been fully fetched (not between `execute()` and `fetchall()`, which invalidates the open cursor on unixODBC), and a multi-statement `CREATE …; SELECT …` walks `cursor.nextset()` to the result set that actually has columns
- FIX: `FullCondition` docstring is now a raw string, so Python no longer emits a `SyntaxWarning: invalid escape sequence '\d'` for the regex example (`^web-\d+\.prod$`) in the Condition Types section
- FIX: `application/modules/rule/models.py` now lints clean — long `condition_types` / `filter_actions` description strings are wrapped, the `match_type` choice list is split across lines, and the file-level `too-few-public-methods` check (false-positive on MongoEngine `EmbeddedDocument` subclasses) is disabled at module scope

## Version 3.12.13
- FEAT: MSSQL/ODBC and MySQL importers optionally permit table-creation in `custom_query` — setting the new `allow_ddl` account flag lets a single `custom_query` do `CREATE TABLE IF NOT EXISTS …; SELECT … FROM …` so the importer can bootstrap its target table. Destructive keywords (DROP, TRUNCATE, DELETE, UPDATE, INSERT, EXEC, GRANT, REVOKE, REPLACE, MERGE) stay blocked even with the flag on, and the query must still contain a SELECT so the importer has rows to iterate
- PERF: "Calculating Hostrules and Attributes" is fast again on large installations — the per-run Checkmk caches that were moved to instance state in the previous release made every multiprocessing worker dispatch pickle the full prefetched host/folder tables, growing pipe traffic quadratically with the host count. The worker pickle now strips the caches the worker does not read (host, folder and bulk queues), cutting per-task pickle payload from kilobytes-per-host to a few hundred bytes — roughly the difference between "stalled" and "done" at 10k-140k-host scale
- FIX: Checkmk rule export no longer re-pushes rules on every run when Checkmk enriches the stored value with schema defaults — the rule comparison now treats stored-value keys that were not part of our configured value as server-side defaults instead of drift, ending the UPDATE/DELETE churn users reported on otherwise untouched rules
- FIX: Jinja-rendered rule values (Checkmk rule conditions, regex patterns, passwords and other API payloads) are no longer HTML-escaped — characters like `&`, `<`, `>`, `'` and `"` are passed through unchanged again, so regex conditions on service descriptions keep matching
- FIX: Checkmk rule export no longer crashes with `KeyError: 'value_template'` after the first matching host — the speed-up in 3.12.12 made rule outcomes shared across hosts, and the export now works on its own copy instead of mutating the shared rule
- FIX: Checkmk inventorize no longer crashes on labels whose name embeds a FQDN (for example `piggyback_source_<host.example.com>`) — dots in label-derived inventory keys are flattened to underscores, matching the HW/SW inventory path
- FIX: MySQL/ODBC custom queries accept CTE (`WITH ... SELECT ...`) statements again — the read-only check no longer rejects valid analytical queries
- FIX: Checkmk downtime sync with weekday offsets that wrap into the next week (for example Friday + 3 days) no longer crashes — the offset now rolls into the following week instead of aborting the downtime run
- FIX: Checkmk host and rule syncs no longer carry state across accounts or repeated runs in one process — version info, host caches, bulk create/update queues, counters, inventorize caches, DCD rules and password IDs all start empty for each sync, preventing wrong cleanup, duplicate create/update actions and stale inventory writes
- FIX: Imported Checkmk sites now receive the same source-account metadata as any other imported object, so they are covered by conflict protection, inventory sync attribution, and account lifecycle cleanup
- FIX: First-time import of a host via the Checkmk v1 importer no longer crashes with `AttributeError` — the initial-add path now calls the existing import-sync helper
- FIX: Password changes and password resets now update the `last_login` audit timestamp correctly (previously written to a non-existent `lastLogin` field, so the audit stayed stale)
- FIX: `POST /api/v1/objects/<hostname>` now returns `403 account_conflict` when a host is claimed by a different source account, and only returns `200 saved` on a successful save — previously it silently returned `200` for every request regardless of the conflict state
- FIX: Admin bulk actions for assigning a CMDB template and a Checkmk version no longer fail with a CSRF error — the forms now include the CSRF token like every other admin form
- FIX: `cmdbsyncer user create_user` re-enables a previously disabled admin account again — a typo (`disable` vs. `disabled`) kept disabled accounts locked after password reset
- FIX: 2FA enrollment form now includes the CSRF token, so saving a new TOTP secret works again with global CSRF protection enabled
- FIX: Plugin upgrades are now atomic — a broken `.syncerplugin` archive no longer deletes the previously installed plugin; the installer extracts into a staging directory and only swaps in the new copy once the archive has been fully unpacked
- FIX: Cached state gate in `StateFile.need_update` works as intended again — a typo (`last_udpate`) made every entry look like "never updated", bypassing the cache
- FIX: Syncer rule automation re-reads host attributes on every run instead of reusing a process-wide cache, so rules are generated from the current database state and not from a stale snapshot
- SEC: Logout is now POST-only with CSRF protection — a GET to `/logout` returns a confirmation page instead of silently ending the session, closing the trivial CSRF-logout vector
- SEC: License information view is restricted to global admins — enterprise load status, registered features and hooks are no longer exposed to every authenticated user
- SEC: Debug views (`debug_host`, `debug`) no longer render the `hostname` query parameter as an unquoted HTML attribute, closing a reflected attribute-injection / XSS vector for authenticated users
- SEC: HTTP plugin debug log now redacts response bodies (access/refresh tokens, passwords, secrets) and omits non-JSON response bodies; JDisc login passes the password as a GraphQL variable instead of embedding it in the query text, so credentials no longer leak into debug logs
- FIX: NetBox Dataflow sync no longer crashes with `KeyError: 'custom_fields'` on field-only updates — the update path only merges `custom_fields` when the payload actually contains custom-field changes

- FIX: Classic on-prem Jira Insight importer now honors the account-level SSL / CA settings (verify_cert, ca_cert_chain, ca_root_cert) like every other plugin, because it routes requests through the shared HTTP path
- FIX: Classic on-prem Jira importer pages through `/environments` and `/servers` until all entries are fetched, so large Jira inventories are imported completely instead of being truncated at the first page
- FIX: Jira Cloud asset import now pages through AQL results until the workspace is exhausted, so large workspaces are imported completely
